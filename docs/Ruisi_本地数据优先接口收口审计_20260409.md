# Ruisi 本地数据优先接口收口审计

## 1. 结论

针对 `qeeshu-ruisi` 当前产品定位：

- 客户业务数据默认保存在本地桌面端
- 云端只负责 `API Key` 鉴权、钱包计费、模型路由与模型调用
- 不再要求客户业务前端直接依赖云端业务主数据接口

因此需要把接口分成三类：

### A. 继续对客户开放的云端接口

这些接口仍适合保留在 `QeeClaw Cloud / Platform API` 中：

- `GET /api/users/me/context`
- `GET /api/platform/models`
- `GET /api/platform/models/providers`
- `GET /api/platform/models/runtimes`
- `POST /api/platform/models/resolve`
- `POST /api/platform/models/route`
- `POST /api/platform/models/invoke`
- `GET /api/platform/models/usage`
- `GET /api/platform/models/cost`
- `GET /api/platform/models/quota`
- `GET /api/billing/wallet`
- `GET /api/billing/records`
- `GET /api/billing/summary`

说明：

- 这一层是“云能力面”
- 主要回答“谁在调用、还能调用多少、该用哪个模型、花了多少钱”
- 可以继续由 `baseUrl + API Key` 直接访问

## 2. 不建议继续对客户开放的云端业务接口

以下接口不符合 `Ruisi` 的“本地数据优先”定位，不建议继续作为客户接入 SDK 的主路径：

### 知识库

- `/api/platform/knowledge/upload`
- `/api/platform/knowledge/list`
- `/api/platform/knowledge/search`
- `/api/platform/knowledge/delete`
- `/api/platform/knowledge/download`
- `/api/platform/knowledge/stats`
- `/api/platform/knowledge/config`
- `/api/platform/knowledge/config/update`

原因：

- 企业知识文件不应先上传云端再回传本地
- 知识索引、切片、检索、删除都应在本地完成

### 会话 / CRM / 销售过程

- `/api/platform/conversations`
- `/api/platform/conversations/stats`
- `/api/platform/conversations/groups`
- `/api/platform/conversations/groups/{roomId}/messages`
- `/api/platform/conversations/history`
- `/api/platform/conversations/messages`
- `/api/platform/approvals/*`
- `/api/workflows/*`

原因：

- 这些都属于客户主业务数据
- 对 `Ruisi` 来说应沉到本地数据库与本地服务层

### 设备 / 通道 / 审计

- `/api/platform/devices/*`
- `/api/platform/channels/*`
- `/api/platform/audit/*`

原因：

- 本地设备状态应来自本机
- 通道绑定、微信接入等如果是本地终端能力，应由本地服务协调
- 审计日志也应优先落本地

### 记忆 / 策略

- `/api/platform/memory/*`
- `/api/platform/policy/*`

原因：

- 如果面向 `Ruisi` 单机客户交付，这些都应先走本地策略与本地知识/记忆
- 云端保留为控制面或未来企业版能力，不适合当前客户前端直接依赖

## 3. 建议改成本地桌面接口的能力域

以下能力域应统一落到 `Electron Main + 本地 SQLite + 本地文件系统`：

- 知识库
- CRM
- 会话中心
- 审计日志
- 设备状态
- 方法论 / 框架
- 草稿 / 写作资产

建议本地服务域：

- `knowledge.*`
- `crm.*`
- `conversation.*`
- `audit.*`
- `device.*`
- `methodology.*`
- `draft.*`

## 4. 当前 Ruisi 已完成的收口

截至 `2026-04-09`，`Ruisi` 已完成以下收口：

- Dashboard：本地
- Search：本地
- Assets：本地
- CRM：本地
- Methodology：本地
- Drafts：本地
- AI 执行日志：本地
- 知识文件上传：本地直传，不再经云端中转

## 5. 当前仍保留的云端调用

当前仍建议保留的 `Ruisi -> Cloud` 调用：

- 解析当前 API Key 对应上下文
- 查询模型与 provider 列表
- 查询额度、用量、成本、钱包余额
- 执行模型推理调用

这部分属于“云能力面”，保留合理。

## 6. 接下来后端重构建议

### 对外客户 API

建议未来客户看到的云端接口只保留：

- `auth/context`
- `models/*`
- `billing/*`

### 云端内部接口

以下建议降级为内部控制面接口，不再作为客户交付 SDK 主文档公开：

- `platform/knowledge/*`
- `platform/conversations/*`
- `platform/devices/*`
- `platform/channels/*`
- `platform/audit/*`
- `platform/memory/*`
- `platform/policy/*`
- `workflows/*`
- `approvals/*`

建议同步提供两套在线文档入口：

- 客户公开版：`/docs/public` 与 `/openapi-public.json`
- 内部控制面版：`/docs/internal` 与 `/openapi-internal.json`

### 本地桌面接口

建议在 `ruisiDesktop` / 本地 gateway 中逐步补齐：

- `knowledge.upload/list/search/delete/reindex`
- `crm.customer.save/list`
- `crm.opportunity.save/list`
- `conversation.thread.list`
- `conversation.message.list/send`
- `audit.record/list`
- `device.status/current`
- `methodology.profile.save/get`

## 7. 最终目标形态

### 云端

- 用户认证
- API Key 管理
- 钱包 / 计费
- 模型路由
- 模型调用

### 本地

- 文件
- 知识库
- 会话
- CRM
- 审计
- 设备状态
- 方法论
- 业务主数据

这才与 `Ruisi` 的桌面交付模式和“赚 tokens 钱”的商业模式一致。
